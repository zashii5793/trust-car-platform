# Trust Car Platform — 事業性評価（辛口レビュー）

> 作成: 2026-06-18 / AIセッションによる自律分析
> 既存の `MARKETING_APPEAL_REPORT.md`（アピール寄り）に対する**批判的カウンターパート**。
> 「夢」ではなく「本当に商売になるか」を実装ベースで評価する。

---

## 0. 結論（先に言う）

**判定: 条件付きで「あり」。ただし全方位同時展開は「なし」。**

このプロダクトは B2C / B2B / B2B2 / C2C を全部やろうとしている。設計としては美しいが、
**スタートアップが4つのマーケットを同時に立ち上げて勝った例はほぼ無い。**
実装の完成度は高い（Service層・課金階層・Firestoreルールまで堅牢）が、
**事業の成否を決めるのはコードではなく「最初のニワトリ・タマゴをどう割るか」**である。

推奨: **個人ユーザーの「車検・整備リマインダー」一点突破で無料ユーザーを貯め、
それを在庫（リード）にして整備工場へ B2B 課金する B2B2C モデルに資源を集中する。**
パーツ C2C（手数料8%）とフリート法人は当面「捨て筋」として凍結を推奨。

## 1. このビジネスの本質的な強み（実装で裏付けあり）

| 強み | 実装根拠 | なぜ効くか |
|------|---------|-----------|
| **データが自動でリードになる構造** | `fleet_service`, `inquiry`, AI提案が車両データ起点 | ユーザーの整備記録＝「今まさに金を使う見込み客」。広告で買うリードより質が高い |
| **「断言しないAI」というポジショニング** | `part_recommendation_service` が常に複数候補+理由+デメリット | カー業界の不信感（押し売り・ぼったくり）への直接的アンチテーゼ。差別化が思想レベルで一貫 |
| **課金インフラが先に出来ている** | RevenueCat統合、4階層shop plan、8%手数料計算 | 多くのアプリが後回しにする収益導線が設計済み。意思決定が速い |
| **乗り換えコストの自己生成** | 整備履歴が貯まるほど離脱しにくい | 記録系アプリ特有のロックイン。継続率の天然の堀になる |

→ **「カーオーナーの不信感」を突く思想と、データ→送客の構造は本物。ここは伸ばす。**

## 2. 辛口：事業として弱い・危ういポイント

### 2.1 マーケットプレイスのニワトリ・タマゴ問題（最大リスク）
- 整備工場は「ユーザーがいる所」にしか金を払わない。ユーザーは「工場が探せる所」にしか来ない。
- **両側ゼロから同時に立ち上げるのは構造的に詰みやすい。**
- 現状コードは「機能は全部ある」が「どちらの side から埋めるか」の戦略が docs に無い。
- **対策**: まず**片側（個人ユーザー無料層）を非マーケット価値（リマインダー単体）で1万人規模まで貯める**。
  工場営業はその後。「あなたの商圏に整備時期のユーザーが◯人います」と言えて初めて B2B が売れる。

### 2.2 B2B の ROI を工場に証明できていない
- スタンダード ¥3,980/月を払う整備工場が知りたいのは唯一**「で、何件成約した？」**。
- 現状、問い合わせ status（pending/replied/closed）は持つが、
  **成約・施工完了・入金までのパイプライン可視化（CRM）が未実装**（調査で確認済）。
- ROI が見えない B2B SaaS は**3ヶ月で解約される**。これは価格設定より重い。
- **対策**: MVPでは「問い合わせ数」だけでも月次でメール通知。`hasMonthlyReport` フラグを最小実装で先に出す。

### 2.3 KYC（事業者認証）不在＝信頼プラットフォームとして致命的
- コンセプトが「信頼（Trust）」なのに、**整備工場の本人確認・営業許可検証が無い**（`Shop.isVerified` はあるが検証フロー未実装）。
- 偽工場が登録→ユーザーがトラブル→「Trust」の看板が逆に毀損。
- **対策**: ローンチ時は**手動審査でよい**（数十店規模なら人力で十分）。自動化は後。ただし「審査済バッジ」のUI表示は最初から必須。

### 2.4 単価 ¥980/月の B2C は「単体では」黒字化しづらい
- カー系アプリの月額課金は継続率が低い（使用頻度が週1未満）。
- `OPERATIONS_COST_ESTIMATE.md` のAI/Firebaseコストを踏まえると、**B2C課金だけで運営費を賄うのは1〜2万人課金規模が必要**で、CACを考えると単独では苦しい。
- **B2Cは「課金で稼ぐ」より「無料で母数を貯めてB2Bの在庫にする」と割り切る方が筋が良い。**

### 2.5 C2Cパーツ手数料8%は今やる事業ではない
- メルカリ/ヤフオク/モノタロウが既に存在。在庫連携も未実装（調査で確認）。
- 8%×低単価パーツ＝**運用コスト割れ確実**。トラブル対応・返品・送金の運用負荷だけ重い。
- **対策**: 凍結。リソースを 2.1〜2.3 に回す。

## 3. ユニットエコノミクスの正気チェック（概算）

> 正確な数字は `OPERATIONS_COST_ESTIMATE.md` 参照。ここは「事業として成立する形」の試算。

**推奨モデル（B2B2C 集中）での損益イメージ:**

| 項目 | 仮定 | 月次インパクト |
|------|------|--------------|
| 無料ユーザー | 10,000人（リード在庫） | コスト中心（AI/DB） |
| 有料工場 | 100店 × 平均¥6,000/月 | **+¥600,000** |
| B2Cプレミアム転換 | 無料の3% = 300人 × ¥980 | +¥294,000 |
| 運用コスト | Firebase+AI+RevenueCat | −¥150,000〜（規模依存） |
| **粗利目安** | | **約 +¥70万/月** |

→ **鍵は「工場100店」**。B2Cは添え物。
   工場1店の LTV（¥6,000×継続24ヶ月＝¥144,000）が CAC（営業1件の獲得コスト）を上回るかが全て。
   **だからこそ B2B には ROI レポート（2.2）が必須**＝解約を止めれば LTV が倍になる。

## 4. 推奨：Go-To-Market の「楔（くさび）」

```
Phase A（0-6ヶ月）: 個人ユーザー獲得 — マーケット機能は「おまけ」
  - 訴求は1つだけ:「車検・点検をAIが忘れさせない」
  - OCRで車検証スキャン→即リマインダー、の摩擦ゼロ体験を磨く
  - KGI: 無料ユーザー 1万 / 整備記録を持つアクティブ率

Phase B（6-12ヶ月）: 1商圏で工場B2Bを手動で立ち上げ
  - 例:「○○市の整備工場10店」に絞って人力営業
  - 売り文句:「市内に整備時期のユーザーが◯人。問い合わせ実績を月次報告」
  - KGI: 有料工場の3ヶ月継続率 70%+（ROIレポートで死守）

Phase C（12ヶ月+）: 横展開 / フリート・C2Cを"必要なら"解凍
```

**「全部入りアプリ」を売るな。「車検を忘れない個人体験」を売り、裏で工場に課金しろ。**

## 5. ローンチ前に潰すべき事業リスク Top3（優先順）

1. **B2B ROIレポートの最小実装**（`hasMonthlyReport`）— 解約防止＝LTV倍増。最優先。
2. **工場の手動KYC＋審査済バッジUI** — 「Trust」の看板を守る最低条件。
3. **個人オンボーディングの摩擦ゼロ化** — OCR→リマインダーまでを最短タップで（今回のUX改善はこの一歩）。

## 6. 今回のセッションでのUI/UX改善（事業仮説との接続）

事業の楔は「個人ユーザーの摩擦ゼロ体験」。今回は**状態表示の明確化と視認性**を地道に改善した:

| 改善 | ファイル | 事業上の意味 |
|------|---------|------------|
| タイムラインのローディングに文脈メッセージ追加 | `vehicle_detail_screen.dart` | 「固まった？」の不安を消し、記録閲覧の継続率を守る |
| 車両情報の値を `Expanded` 化（長い値のはみ出し防止） | `vehicle_detail_screen.dart` | 車検満了日など重要情報の表示崩れを防止 |
| 提案バッジのダークモード・コントラスト改善 | `home_screen.dart` | AI提案（=送客起点）の視認性向上＝CV導線の保護 |

> 大規模リファクタ（AppButton/AppDialog統一・採用率30%問題）は別Issue化を推奨。
> 詳細な改善候補は本セッションの調査メモ参照。
