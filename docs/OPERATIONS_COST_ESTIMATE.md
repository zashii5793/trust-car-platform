## スケーリング時の注意事項

### Firestoreインデックスコスト

複合インデックスが増えるとスキャンコストが上昇。以下を監視:
- `maintenance_records` の `vehicleId + type + date` 複合クエリ
- `community_maintenance_trends` の `maker + model` 参照
- `faqs` の `category + createdAt` ソート

### Cloud Storageの画像容量

- ナンバープレートマスキング後の画像は PNG で再保存 → 元画像より大きくなる可能性
- 圧縮率改善: JPEG 品質 70 で保存 (現行実装) → Storage $60/月 (100K MAU) は妥当

### AI コスト最適化

- 整備コメント: Rule-based で対応できる場合はAI不使用 (MaintenanceCommentService)
- 車種傾向分析: CommunityTrendService は Firestore 集計値のみ → AI 不使用
- 問い合わせ返答案: AI 使用 → レート制限 20回/日を維持

## 法的リスクと運用コスト

| リスク項目 | 対策 | 追加コスト |

| 安全運転情報の責任 | JAF/警察公式へのリンクのみ、ユーザー投稿を「参考」明示 | $0 |

## 推奨アーキテクチャ変更 (コスト最適化)

1. **読み取り最適化**: オンメモリキャッシュ (`Provider` レベル) → Firestore 読み取り 30% 削減
2. **画像最適化**: アップロード前 JPEG 70 圧縮 (実装済み) → Storage 40% 削減
3. **Cloud Functions 統合**: RevenueCat webhook + 通知 + 集計を1関数化 → コールド起動削減
4. **Firestore セキュリティルール強化**: クライアント直接書き込み削減 → Function 経由に統一
